## P1-1 — Engine never detects broker-side take-profit fills

**Severity:** P1 (blocking)

**Issue:** The bot's winning trades hit the broker-side TP and close on the broker. The engine continues to think the position is open. It will eventually:
1. Fire the 45-minute stale-position watchdog → CRITICAL log + alert + `_force_exit(bar.close, "STALE_POSITION_WATCHDOG")` + halt-until-review. The recorded exit price is bar.close at watchdog time, NOT the TP price (~$30-100 mis-attribution per trade typical). The day is then halted for any further entries.
2. Or, on a quiet rest of the day, the 15:30 force-EOD fires `_force_exit(bar.close, "TIME_EXIT")` — same mis-attribution problem.

**Files / lines (redacted):**
- `src/live/webhook_engine.py:80-97` (Position class — only stores `protective_stop_price`, not `abs_target`)
- `src/live/webhook_engine.py:1185-1194` (entry path drops `result["abs_target"]` on the floor)
- `src/live/webhook_engine.py:1617-1706` (`_check_phantom_position` only checks stop-level breach, not target-level)
- `src/live/webhook_engine.py:776-785` (`_broker_position_sync` is gated on `trail_source in ("broker","both")`)
- `src/live/<broker>_client.py:226-235` (3-leg OCO TP IS placed correctly on the broker)

**Reproduction:** With `take_profit_pts: 150` and `trail_source: bot`: enter long, broker fills TP, bot still tracks position. By 45 minutes later, the stale watchdog fires synthetic exit at bar.close recording roughly half the realized P&L instead of the actual TP price.

**Fix sketch:** Add `target_price=result.get("abs_target")` to the `Position` constructor and store it on the Position. Extend phantom-detection to also check `bar.high >= pos.target_price` (long) or `bar.low <= pos.target_price` (short). When triggered, exit at the target price with reason `TP_FILL` and DO NOT halt — TP is the expected winner path, not a phantom event.

## P1-2 — Hardcoded CT→ET offset breaks at DST transitions

**Severity:** P1 (dormant — fires only across DST boundaries)

**Issue:** Six sites do `bar_dt.hour + 1` or `bar_dt + timedelta(hours=1)` to convert IB Gateway's CT bar timestamps to ET. The offset assumes:
1. IB returns naive Central Time bars
2. The gap between CT and ET is always exactly 1 hour

Both assumptions break at DST seams. The engine's wall-clock (`now = datetime.now(ET)`) is timezone-aware DST-correct, while the bar arithmetic is naive — so during the spring/fall transition weekend, the bar arithmetic and wall-clock will be 1 hour apart for one bar, and the 09:30-10:00 entry window will be off by an hour for that one Sunday→Monday seam.

Worse: if `ib_insync` returns tz-aware datetimes (depends on Gateway version + `formatDate` setting), `bar_dt.hour` is already in ET and adding 1 silently shifts everything by an hour permanently — producing 16:30 bar_times for 15:30 closes, breaking force-EOD.

**Files / lines (redacted):**
- `src/live/webhook_engine.py:639-646` (bar_close_hour = bar_dt.hour + 1)
- `src/live/webhook_engine.py:653, 663` (current_et_date = (bar_dt + timedelta(hours=1)).date())
- `src/live/range_filter.py:40-45` and `52-54`
- `src/live/efficiency_filter.py:49-54` and `67-70`

**Fix sketch:** Replace all six sites with:
```python
bar_dt_et = (
    bar_dt if bar_dt.tzinfo
    else pytz.timezone("US/Central").localize(bar_dt)
).astimezone(ET)
```
Lock IB Gateway `formatDate`/`formatTime` in `connect()` so the assumption is enforced. Add a regression test that feeds a tz-aware ET-localized bar and asserts `bar_time` is computed correctly.

## P2-1 — Discord alert is a non-awaited coroutine, never sent

**Severity:** P2

**File / line:** `src/live/ib_streamer.py:124-128`

**Issue:** The line
```python
self._discord_alert(f":rotating_light: {msg}")
```
returns a coroutine that is discarded. The alert never fires. The whole purpose of this branch (last-resort CRITICAL escalation before raising to systemd) is silently broken. You'll get a `RuntimeWarning: coroutine ... was never awaited` if Python is configured to surface them, but no Discord ping. This is the exact condition the operator wants visibility on (5+ failed connect attempts → IB Gateway is broken).

**Reproduction:** Block the IB Gateway port via firewall, start the bot. After ~5 minutes of failed connects, the CRITICAL log fires but Discord stays silent.

**Fix sketch:** One character.
```python
await self._discord_alert(f":rotating_light: {msg}")
```

This is a tiny finding by line count, but it disables the most important alert in the system. Auditors notice these because they read every async path adversarially: "is the coroutine actually awaited?" "is the cancellation propagated?" "is the timeout enforced?" Generic code review tools rarely catch this pattern.
